- Launch Mission Planner.

	If SITL is already running, MissionPlanner will connect to it via UDP automatically,
	otherwise, use the drop-down in the top right corner and enter '14550' for port number.

	Switch to the 'PLAN' tab to read waypoints stored in the EEPROM of the aircraft, or specify new path.
	Make sure to select the "TAKEOFF" command for waypoint #1, 'Frame' should be set to "Relative" for all waypoints.
	Select the "LAND" command for the last waypoint, but this is not strictly necessary

	When finished with planning the new mission, click "Write" on the panel to the right.

	Switch back to the "DATA" tab and go to "Actions". The aircraft should now be in "Stabilize" mode.
	To arm the aircraft, click the "Arm/Disarm" button, use "Force Arm" if necessary.
	To take off, select "Mission_Start" from the drop-down next to the "Do Action" button, click "Do Action"
	and "Yes" in the next dialogue.

	At this point the simulated aircraft should take off and start executing the mission.


- Run the Python script named "mexico_gcs_script_satcom_main.py" in PowerShell

	Make sure that the parameters in the file config.json are correct and that CloudLoop account is accessible
	via the web interface. Use Mert's email address as login name and Malloy123456! as password.

	Windows PowerShell:

	$ python mexico_gcs_script_satcom_main.py

	This script starts a UDP server on port 16000, periodically checks the CloudLoop mailbox for incoming high-latency
	MAVLink messages from the aircraft and relays them to the GCS. It also takes MAVLink messages from the GCS and sends
	them to the CloudLoop server, where the aircraft can access them using the Iridium modem.


- Launch QGC (make sure to use "Malloy QGC Mex-v1.0.11")

	Click the gear icon in the top left corner and then "Application Settings"
	Choose "Comm Links" on the left and click "Add".
	Enter a name for the new link, tick the box next to "High Latency" and pick "UDP" from the "Type" drop-down.
	Enter "16000" for port and save the link by clicking "OK".

	Once the link is created, click "Connect" on it and go back to the main screen.

	At this point we should see the position and heading of the simulated aircraft on the map, the SITL default location
	is an airfield just outside Canberra, Australia.


- Run the Python script named "waypoint_test.py" in PowerShell

	The script sends a COMMAND_LONG (76) MAVLink message with the command MAV_CMD_DO_SET_MODE to the aircraft which is an
	instruction to switch to "Guided" mode. The aircraft needs to be in this flight mode to be able to accept a new waypoint.
	The script sends the new waypoint after a 5 second delay, using the SET_POSITION_TARGET_GLOBAL_INT MAVLink message.

	The expected outcome is that upon receiving the MAV_CMD_DO_SET_MODE command, the aircraft stops executing the mission, and
	on receiving SET_POSITION_TARGET_GLOBAL_INT, starts flying to the new waypoint in a straight line.

	Switching back to "Auto" mode, will result in the aircraft resuming the mission by flying to the waypoint which was going to
	be the next to reach had the mission not been aborted by the MAVLink messages sent by the Python script.


	Progress so far (06/08/2025):

	When set up correctly, sim_vehicle.py runs without errors, MissionPlanner connects to SITL automatically, the aircraft can
	take off, execute the mission, land, etc.
	The Lua script successfully detects the Iridium9603 modem and starts sending HIGH_LATENCY2 (235) messages to CloudLoop, where
	QGC combined with the Python script mexico_gcs_script_satcom_main.py can read these messages and display the position and heading
	of the aircraft on the map.
	The Lua script does receive messages sent by the script waypoint_test.py, but at present parsing the messages isn't working correctly.

	When the script waypoint_test.py sent the messages via TCP instead of the CloudLoop API at a previous stage of testing, it was found
	that both messages had to be sent twice for them to take effect, it is yet to be determined whether this issue persists when switching
	to CloudLoop. If it does, it needs to be confirmed that both messages arrive to the aircraft and if so the Lua script should be tweaked
	to keep calling the appropriate methods on the vehicle inside a loop as many times as necessary to achieve the expected outcome.
